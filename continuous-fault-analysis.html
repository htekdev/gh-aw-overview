What is Continuous Fault Analysis?

CI failures are one of the biggest productivity drains in software engineering. A developer pushes code, the build breaks, and then they spend 20 minutes scrolling through log output trying to find the relevant error buried in thousands of lines. Continuous Fault Analysis automates this: when a workflow run fails, an AI agent reads the logs, correlates the failure with the code changes that triggered it, identifies the root cause, and posts a clear explanation — optionally with a draft PR containing the fix.

📋

Automated Log Parsing

The agent reads the full CI log output, filters noise, and extracts the actual error messages, stack traces, and failing test names — saving developers from scrolling through thousands of lines.

🔗

Change Correlation

Cross-references the failure with the commit diff that triggered the run. The agent identifies which specific code changes likely caused the failure and explains the connection.

💡

Root Cause Explanation

Posts a human-readable explanation of why the build failed, what the error means, and what the developer should look at — written in plain English, not log-speak.

🔧

Suggested Fix PRs

For common failure patterns (missing imports, type errors, config issues), the agent can optionally open a draft PR with the proposed fix for the developer to review.

Example Workflow

A fault analysis workflow that triggers on CI failures and investigates the root cause.

📄 .github/workflows/fault-analysis.md 
---
name: CI Fault Analysis
description: Investigate CI failures and explain root cause

on:
  workflow_run:
    workflows: ["CI", "Tests", "Build"]
    types: [completed]
    branches: [main]

permissions:
  contents: read
  actions: read
  issues: read
  pull-requests: read

safe-outputs:
  create-issue:
    title-prefix: "[ci-failure] "
    labels:
      - ci-failure
      - needs-investigation
    max: 1

tools:
  github:
---

# CI Fault Analysis

You are a CI/CD debugging expert for
${{ github.repository }}.

A workflow run has just completed. If it
failed, investigate:

1. Download and read the workflow run logs
2. Identify the specific error messages,
   failing tests, or build errors
3. Examine the commit(s) that triggered
   this run and their diffs
4. Correlate the changes with the failures
5. Determine the root cause

Create an issue with:
- A clear summary of what failed
- The root cause explanation
- The relevant code changes
- A suggested fix or next steps

If the run succeeded, do nothing.

Security Considerations

  • workflow_run trigger: Fires after the CI workflow completes, not during — the agent never interferes with the actual build
  • actions: read: Allows the agent to download workflow run logs but not modify or re-trigger runs
  • Conditional execution: The prompt instructs the agent to do nothing on success — no unnecessary compute
  • Single issue output: The agent can create at most one issue per failure, preventing noise
  • No write access to code: The agent cannot push fixes directly; it can only report findings

Anatomy of a Fault Analysis Report

1
Failure Summary: "Tests failed in `auth/login.test.ts` — 3 assertions failed in the session validation suite"
2
Root Cause: "Commit `a3f2b1c` changed the session token format from JWT to opaque tokens, but the test fixtures still use JWT format"
3
Relevant Code: Links to the specific lines in the diff that caused the mismatch
4
Suggested Fix: "Update test fixtures in `auth/__fixtures__/` to use the new opaque token format"

Advanced: Auto-Fix PRs

For teams that want the agent to go beyond reporting and actually propose fixes, add create-pull-request to safe-outputs.

📄 Extended safe-outputs for auto-fix 
# Add to the frontmatter to enable fix PRs
safe-outputs:
  create-issue:
    title-prefix: "[ci-failure] "
    labels: [ci-failure]
    max: 1
  create-pull-request:
    title-prefix: "[auto-fix] "
    labels: [auto-fix, needs-review]
    max: 1
    draft: true

When to Use Auto-Fix PRs

Auto-fix PRs work best for deterministic failure patterns: missing imports, type mismatches, outdated test fixtures, and configuration drift. For complex logic bugs, the issue-only approach is safer — the agent explains the problem and lets a human write the fix. Always set draft: true so fixes require explicit human approval before merging.
