What is Continuous Quality?

Linters catch syntax issues. Static analysis catches type errors. But neither can tell you that a function is poorly named, that a module has grown too complex, that error handling is inconsistent across the codebase, or that the code doesn't follow the team's architectural patterns. Continuous Quality uses an AI agent to perform deep, semantic code review on a schedule — analyzing code quality, suggesting improvements, and ensuring adherence to coding standards that are too nuanced for rule-based tools.

Example Workflow

A daily code quality scan that identifies improvements and delivers them as draft PRs.

📄 .github/workflows/continuous-quality.md 
---
name: Continuous Code Quality
description: Daily code quality analysis and improvement

on:
  schedule:
    - cron: "0 6 * * 1-5"  # Weekdays 6am UTC
  workflow_dispatch:

permissions:
  contents: read
  pull-requests: read

safe-outputs:
  create-pull-request:
    title-prefix: "[quality] "
    labels:
      - code-quality
      - refactor
      - automated
    max: 1
    draft: true

tools:
  github:
  edit:
---

# Continuous Code Quality

You are a senior code reviewer for
${{ github.repository }}.

Perform a daily code quality review:

1. Scan the repository for code quality
   issues, focusing on recently changed files
2. Identify ONE focused improvement area:
   - Overly complex functions (simplify)
   - Inconsistent error handling (standardize)
   - Missing or outdated code comments
   - Dead code or unused imports
   - Naming that could be clearer
   - Duplicated logic (DRY violations)
3. Make the improvement directly in code
4. Create a draft PR with:
   - Clear description of what was improved
   - Why the change improves quality
   - Before/after comparison if helpful

Rules:
- ONE focused change per PR (not a kitchen sink)
- Never break existing tests
- Preserve existing functionality
- Follow the project's existing style

Design Philosophy: Small, Focused PRs

  • One improvement per run: The prompt explicitly says "ONE focused improvement area" — this prevents the agent from creating a massive, unreviewable PR
  • Draft PRs only: draft: true ensures a human must explicitly approve and merge every change
  • edit tool: The agent needs the edit tool to modify files, but all changes are buffered through safe-outputs
  • No test breakage: The prompt instructs the agent to preserve functionality and never break tests
  • Style preservation: The agent follows the project's existing conventions, not its own preferences

The Compound Effect

One small refactor per day may seem modest, but over time the effect compounds. In a month, that's ~20 focused quality improvements — dead code removed, naming clarified, error handling standardized — all reviewed and approved by humans. It's like having a junior developer whose only job is to clean up the codebase, one PR at a time, every single day.

How AI-powered quality analysis complements (not replaces) your existing toolchain.

Getting Started

Start with a conservative configuration and expand as you build confidence.

1️⃣

Week 1: Report Only

Start with create-issue instead of create-pull-request. Let the agent identify quality issues and report them without making changes. Review the reports to calibrate.

2️⃣

Week 2: Draft PRs

Switch to create-pull-request with draft: true. Review every PR carefully. Merge the good ones, close the bad ones — the agent learns from the codebase patterns.

3️⃣

Week 3+: Tune the Prompt

Refine the prompt to focus on your team's specific pain points. Add your coding standards document as context. Adjust the schedule to match your team's review capacity.
