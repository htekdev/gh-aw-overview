What is Continuous Triage?

Every open-source maintainer and engineering team knows the pain: issues pile up, labels are inconsistent, duplicates go unnoticed, and new contributors wait days for a first response. Continuous Triage uses an AI agent to read every new issue the moment it arrives, understand its intent, apply the right labels, detect duplicates, and post a helpful first response — all within seconds, all defined in a single Markdown file.

🧠

Context-Aware Labeling

The agent reads the issue body, title, and referenced files to apply labels based on semantic understanding — not keyword matching. It knows the difference between a "bug in the auth module" and a "feature request for auth."

🔍

Duplicate Detection

Before responding, the agent searches existing open and recently closed issues to flag potential duplicates, linking them for the maintainer to confirm or dismiss.

💬

Welcoming First Response

New contributors get an immediate, helpful comment acknowledging their issue, asking clarifying questions if needed, and pointing them to relevant docs or contributing guides.

📊

Priority Assessment

Based on the issue content and affected components, the agent suggests a priority level (P0–P3) so the team can focus on what matters most.

Example Workflow

A complete Continuous Triage workflow in Markdown. Copy this into your .github/workflows/ directory and compile with gh aw compile.

📄 .github/workflows/continuous-triage.md 
---
name: Continuous Issue Triage
description: Auto-triage new issues with labels and response

on:
  issues:
    types: [opened]

permissions:
  contents: read
  issues: read

safe-outputs:
  add-labels:
    allowed:
      - bug
      - enhancement
      - question
      - documentation
      - duplicate
      - good-first-issue
  create-comment:
    max: 1

tools:
  github:
---

# Continuous Issue Triage

You are an expert issue triager for the
repository ${{ github.repository }}.

When a new issue is opened:

1. Read the issue title and body carefully
2. Search existing open issues for duplicates
3. Examine the repository structure and recent
   changes for relevant context
4. Apply the most appropriate label(s)
5. Post a helpful, welcoming comment that:
   - Acknowledges the issue
   - Asks clarifying questions if needed
   - Links to relevant docs or code
   - Suggests next steps

Be concise, friendly, and professional.
Never fabricate information about the codebase.

Why This Works

  • Trigger: on: issues: types: [opened] — fires only on new issues, not edits or comments
  • Read-only permissions: The agent can read contents and issues but cannot write directly
  • Constrained labels: Only the 6 labels in the allowed list can be applied — the agent cannot invent new ones
  • Single comment: max: 1 prevents the agent from spamming the issue thread
  • No secrets exposed: The agent runs in a sandboxed container with zero access to repository secrets

Real-World Impact

In GitHub's own Agent Factory, the issue triage workflow is the most commonly deployed agentic workflow. It achieves consistent labeling accuracy across repositories and reduces the median first-response time from hours to under 30 seconds — a critical metric for open-source community health.

Get Started in 3 Minutes

Set up Continuous Triage in your repository right now.

Terminal 
# 1. Install the gh-aw CLI extension
$ gh extension install github/gh-aw

# 2. Initialize agentic workflows in your repo
$ gh aw init

# 3. Create the triage workflow markdown file
$ cp continuous-triage.md .github/workflows/

# 4. Compile to a hardened .lock.yml
$ gh aw compile .github/workflows/continuous-triage.md

# 5. Push both files and watch it work
$ git add .github/workflows/ && git commit -m "Add continuous triage" && git push
