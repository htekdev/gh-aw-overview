<div class="section-hdr" id="triage">
  <h2>🏷️ Continuous Triage</h2>
</div>

Issues pile up, labels are inconsistent, duplicates go unnoticed, and new contributors wait days for a first response. Continuous Triage uses an AI agent to read every new issue the moment it arrives, understand its intent, apply the right labels, detect duplicates, and post a helpful first response — all within seconds.

<div class="card-grid-2col">
  <div class="card card-accent-green">
    <h3>🧠 Context-Aware Labeling</h3>
    <p class="desc">The agent reads the issue body, title, and referenced files to apply labels based on semantic understanding — not keyword matching.</p>
  </div>
  <div class="card card-accent-blue">
    <h3>🔍 Duplicate Detection</h3>
    <p class="desc">Before responding, the agent searches existing open and recently closed issues to flag potential duplicates.</p>
  </div>
  <div class="card card-accent-orange">
    <h3>💬 Welcoming First Response</h3>
    <p class="desc">New contributors get an immediate, helpful comment acknowledging their issue and pointing them to relevant docs.</p>
  </div>
  <div class="card card-accent-cyan">
    <h3>📊 Priority Assessment</h3>
    <p class="desc">Based on content and affected components, the agent suggests a priority level (P0–P3) so the team focuses on what matters.</p>
  </div>
</div>

### How It Works

<div class="flow-diagram">
  <div class="flow-step"><span class="step-icon">📝</span><span class="step-label">Issue Opened</span><span class="step-desc">Trigger fires</span></div>
  <span class="flow-arrow">→</span>
  <div class="flow-step"><span class="step-icon">🤖</span><span class="step-label">Agent Reads</span><span class="step-desc">Parses title, body, context</span></div>
  <span class="flow-arrow">→</span>
  <div class="flow-step"><span class="step-icon">🔎</span><span class="step-label">Research</span><span class="step-desc">Searches repo & issues</span></div>
  <span class="flow-arrow">→</span>
  <div class="flow-step"><span class="step-icon">🏷️</span><span class="step-label">Classify</span><span class="step-desc">Labels & priority</span></div>
  <span class="flow-arrow">→</span>
  <div class="flow-step"><span class="step-icon">💬</span><span class="step-label">Respond</span><span class="step-desc">Posts via safe-outputs</span></div>
</div>

### Example Workflow

```yaml
# File: .github/workflows/continuous-triage.md
---
name: Continuous Issue Triage
description: Auto-triage new issues with labels and response

on:
  issues:
    types: [opened]

permissions:
  contents: read
  issues: read

safe-outputs:
  add-labels:
    allowed: [bug, enhancement, question, documentation, duplicate, good-first-issue]
  create-comment:
    max: 1

tools:
  github:
---

# Continuous Issue Triage

You are an expert issue triager for this repository.

When a new issue is opened:
1. Read the issue title and body carefully
2. Search existing open issues for duplicates
3. Examine the repository structure for relevant context
4. Apply the most appropriate label(s)
5. Post a helpful, welcoming comment that acknowledges the issue,
   asks clarifying questions if needed, and suggests next steps.
```

<div class="callout callout-green">
  <p>💡 <strong>Why this works:</strong> <code>on: issues: types: [opened]</code> fires only on new issues. Labels are constrained to the <code>allowed</code> list — the agent cannot invent new ones. <code>max: 1</code> prevents comment spam. The agent runs with read-only permissions in a sandboxed container.</p>
</div>

<div class="info-box purple">
  <h4>📌 Production Case Study: GitHub Accessibility Team</h4>
  <p>GitHub's own Accessibility team uses Continuous Triage in production. With 90% of their feedback flowing through a single channel, they built an event-driven pipeline where Copilot analyzes every new issue instantly. The agent automatically populates over <strong>40 data points</strong> (issue type, user segment, affected components, WCAG criteria) and handles <strong>80% of the metadata</strong> automatically. It turns a chaotic backlog into continuous, rapid resolutions.</p>
</div>

<div class="section-hdr" id="summarization">
  <h2>📊 Continuous Summarization</h2>
</div>

Software projects generate a constant stream of activity — commits, PRs, issues, discussions, releases, CI runs. No single person can track it all. Continuous Summarization uses an AI agent on a schedule to read all recent activity, distill it into a concise report, and publish it as a GitHub issue.

<div class="card-grid-2col">
  <div class="card card-accent-blue">
    <h3>📰 Daily Digest Reports</h3>
    <p class="desc">Summaries of the last 24 hours: new issues, merged PRs, CI status, release activity, and contributor highlights — delivered as a GitHub issue every morning.</p>
  </div>
  <div class="card card-accent-green">
    <h3>📈 Trend Analysis</h3>
    <p class="desc">Tracks patterns over time: increasing bug reports in a module, slowing PR review velocity, growing test failures — surfacing trends before they become problems.</p>
  </div>
  <div class="card card-accent-orange">
    <h3>🎯 Actionable Next Steps</h3>
    <p class="desc">Every summary ends with concrete recommendations: stale PRs needing review, issues needing triage, blockers needing attention.</p>
  </div>
  <div class="card card-accent-cyan">
    <h3>🔄 Auto-Close Older Reports</h3>
    <p class="desc">Previous reports are automatically closed when a new one is created via <code>close-older-issues</code>, keeping the issue tracker clean.</p>
  </div>
</div>

### How It Works

<div class="flow-diagram">
  <div class="flow-step"><span class="step-icon">⏰</span><span class="step-label">Cron Trigger</span><span class="step-desc">Daily at 9am UTC</span></div>
  <span class="flow-arrow">→</span>
  <div class="flow-step"><span class="step-icon">📥</span><span class="step-label">Gather Data</span><span class="step-desc">Issues, PRs, commits, CI</span></div>
  <span class="flow-arrow">→</span>
  <div class="flow-step"><span class="step-icon">🧠</span><span class="step-label">Analyze</span><span class="step-desc">Trends & highlights</span></div>
  <span class="flow-arrow">→</span>
  <div class="flow-step"><span class="step-icon">📝</span><span class="step-label">Generate</span><span class="step-desc">Structured report</span></div>
  <span class="flow-arrow">→</span>
  <div class="flow-step"><span class="step-icon">📬</span><span class="step-label">Publish</span><span class="step-desc">Issue via safe-outputs</span></div>
</div>

### Example Workflow

```yaml
# File: .github/workflows/daily-summary.md
---
name: Daily Project Summary
description: Generate a daily digest of repo activity

on:
  schedule:
    - cron: "0 9 * * 1-5"   # Weekdays 9am UTC
  workflow_dispatch:

permissions:
  contents: read
  issues: read
  pull-requests: read

safe-outputs:
  create-issue:
    title-prefix: "[daily-summary] "
    labels: [report, daily-summary]
    close-older-issues: true

tools:
  github:
---

# Daily Project Summary

Create a concise daily status report covering:

1. **Activity** — Issues opened/closed, PRs merged/opened, commits
2. **CI/CD Health** — Workflow run results, failures, flaky tests
3. **Highlights** — Notable achievements, approaching milestones
4. **Action Items** — PRs needing review, stale issues, blockers

Use tables and lists. Keep it concise and actionable.
```

<div class="callout callout-green">
  <p>💡 <strong>Key design decisions:</strong> <code>close-older-issues: true</code> auto-closes yesterday's report. The <code>[daily-summary]</code> prefix makes reports easy to filter. <code>workflow_dispatch</code> allows on-demand runs.</p>
</div>

<div class="section-hdr" id="fault-analysis">
  <h2>🔥 Continuous Fault Analysis</h2>
</div>

CI failures are one of the biggest productivity drains in software engineering. A developer pushes code, the build breaks, and they spend 20 minutes scrolling through logs trying to find the error. Continuous Fault Analysis automates this: when a workflow run fails, the agent reads the logs, correlates the failure with the code changes, identifies the root cause, and posts a clear explanation.

<div class="card-grid-2col">
  <div class="card card-accent-orange">
    <h3>📋 Automated Log Parsing</h3>
    <p class="desc">The agent reads full CI output, filters noise, and extracts actual error messages, stack traces, and failing test names.</p>
  </div>
  <div class="card card-accent-blue">
    <h3>🔗 Change Correlation</h3>
    <p class="desc">Cross-references the failure with the commit diff. Identifies which specific code changes likely caused the failure.</p>
  </div>
  <div class="card card-accent-green">
    <h3>💡 Root Cause Explanation</h3>
    <p class="desc">Posts a human-readable explanation of why the build failed — in plain English, not log-speak.</p>
  </div>
  <div class="card card-accent-cyan">
    <h3>🔧 Suggested Fix PRs</h3>
    <p class="desc">For common patterns (missing imports, type errors, config issues), optionally opens a draft PR with the proposed fix.</p>
  </div>
</div>

### How It Works

<div class="flow-diagram">
  <div class="flow-step"><span class="step-icon">❌</span><span class="step-label">CI Fails</span><span class="step-desc">workflow_run completed</span></div>
  <span class="flow-arrow">→</span>
  <div class="flow-step"><span class="step-icon">📜</span><span class="step-label">Read Logs</span><span class="step-desc">Download & parse output</span></div>
  <span class="flow-arrow">→</span>
  <div class="flow-step"><span class="step-icon">🔍</span><span class="step-label">Correlate</span><span class="step-desc">Diff vs. error analysis</span></div>
  <span class="flow-arrow">→</span>
  <div class="flow-step"><span class="step-icon">🧠</span><span class="step-label">Diagnose</span><span class="step-desc">Root cause identification</span></div>
  <span class="flow-arrow">→</span>
  <div class="flow-step"><span class="step-icon">📝</span><span class="step-label">Report</span><span class="step-desc">Issue or PR via safe-outputs</span></div>
</div>

### Example Workflow

```yaml
# File: .github/workflows/fault-analysis.md
---
name: CI Fault Analysis
description: Investigate CI failures and explain root cause

on:
  workflow_run:
    workflows: ["CI", "Tests", "Build"]
    types: [completed]
    branches: [main]

permissions:
  contents: read
  actions: read
  issues: read
  pull-requests: read

safe-outputs:
  create-issue:
    title-prefix: "[ci-failure] "
    labels: [ci-failure, needs-investigation]
    max: 1

tools:
  github:
---

# CI Fault Analysis

You are a CI/CD debugging expert. A workflow run has just completed.
If it failed, investigate:

1. Download and read the workflow run logs
2. Identify the specific error messages and failing tests
3. Examine the commit(s) that triggered this run
4. Correlate the changes with the failures
5. Determine the root cause

Create an issue with a clear summary, root cause explanation,
relevant code changes, and suggested fix. If the run succeeded, do nothing.
```

<div class="callout callout-green">
  <p>💡 <strong>Security:</strong> The <code>workflow_run</code> trigger fires <em>after</em> CI completes — the agent never interferes with the build. <code>actions: read</code> allows log download but not re-triggering runs.</p>
</div>

### Advanced: Auto-Fix PRs

For teams that want the agent to propose fixes directly, add `create-pull-request` to safe-outputs:

```yaml
safe-outputs:
  create-issue:
    title-prefix: "[ci-failure] "
    labels: [ci-failure]
    max: 1
  create-pull-request:
    title-prefix: "[auto-fix] "
    labels: [auto-fix, needs-review]
    max: 1
    draft: true
```

<div class="callout callout-orange">
  <p>⚠️ Auto-fix PRs work best for deterministic failure patterns: missing imports, type mismatches, outdated test fixtures, and config drift. For complex logic bugs, issue-only is safer — let a human write the fix. Always set <code>draft: true</code>.</p>
</div>

<div class="section-hdr" id="quality">
  <h2>✅ Continuous Quality</h2>
</div>

Linters catch syntax issues. Static analysis catches type errors. But neither can tell you that a function is poorly named, that a module has grown too complex, or that error handling is inconsistent across the codebase. Continuous Quality uses an AI agent on a schedule to perform deep, semantic code review — catching what rule-based tools miss.

<div class="card-grid-2col">
  <div class="card card-accent-green">
    <h3>📏 Coding Standards</h3>
    <p class="desc">Naming conventions, architectural patterns, and style consistency across the codebase.</p>
  </div>
  <div class="card card-accent-blue">
    <h3>🧩 Complexity Reduction</h3>
    <p class="desc">Function length, nesting depth, coupling — the agent simplifies overly complex code.</p>
  </div>
  <div class="card card-accent-orange">
    <h3>🛡️ Error Handling</h3>
    <p class="desc">Consistency, coverage, edge cases — standardize error handling patterns across modules.</p>
  </div>
  <div class="card card-accent-cyan">
    <h3>📖 Documentation Gaps</h3>
    <p class="desc">Missing docs, outdated comments, unexplained logic — the agent writes what's missing.</p>
  </div>
</div>

### How It Works

<div class="flow-diagram">
  <div class="flow-step"><span class="step-icon">⏰</span><span class="step-label">Schedule</span><span class="step-desc">Daily or weekly cron</span></div>
  <span class="flow-arrow">→</span>
  <div class="flow-step"><span class="step-icon">📂</span><span class="step-label">Scan Code</span><span class="step-desc">Read repo structure</span></div>
  <span class="flow-arrow">→</span>
  <div class="flow-step"><span class="step-icon">🧠</span><span class="step-label">Analyze</span><span class="step-desc">Quality assessment</span></div>
  <span class="flow-arrow">→</span>
  <div class="flow-step"><span class="step-icon">✏️</span><span class="step-label">Refactor</span><span class="step-desc">Apply improvements</span></div>
  <span class="flow-arrow">→</span>
  <div class="flow-step"><span class="step-icon">🔀</span><span class="step-label">Draft PR</span><span class="step-desc">Human review required</span></div>
</div>

### Example Workflow

```yaml
# File: .github/workflows/continuous-quality.md
---
name: Continuous Code Quality
description: Daily code quality analysis and improvement

on:
  schedule:
    - cron: "0 6 * * 1-5"   # Weekdays 6am UTC
  workflow_dispatch:

permissions:
  contents: read
  pull-requests: read

safe-outputs:
  create-pull-request:
    title-prefix: "[quality] "
    labels: [code-quality, refactor, automated]
    max: 1
    draft: true

tools:
  github:
  edit:
---

# Continuous Code Quality

You are a senior code reviewer. Perform a daily quality review:

1. Scan the repository, focusing on recently changed files
2. Identify ONE focused improvement area:
   - Overly complex functions (simplify)
   - Inconsistent error handling (standardize)
   - Missing or outdated code comments
   - Dead code or unused imports
   - Naming that could be clearer
3. Make the improvement directly in code
4. Create a draft PR with a clear description of what and why

Rules: ONE focused change per PR. Never break existing tests.
Preserve functionality. Follow the project's existing style.
```

<div class="callout callout-green">
  <p>💡 <strong>The compound effect:</strong> One small refactor per day = ~20 focused quality improvements per month. Dead code removed, naming clarified, error handling standardized — all reviewed and approved by humans. Like having a developer whose only job is to clean up the codebase, one PR at a time.</p>
</div>

### Continuous Quality vs. Traditional Tools

| Capability | Linters / Static Analysis | Continuous Quality (Agentic) |
|---|---|---|
| Syntax & formatting | ✅ Excellent | Unnecessary (use linters) |
| Type errors | ✅ Excellent | Unnecessary (use type checker) |
| Naming quality | ❌ Cannot assess | ✅ Semantic understanding |
| Architectural patterns | ❌ Cannot assess | ✅ Context-aware |
| Code simplification | ⚠️ Basic (cyclomatic) | ✅ Rewrites with intent |
| Documentation gaps | ⚠️ Missing doc warnings | ✅ Writes the docs |
| Dead code detection | ⚠️ Partial | ✅ Removes with context |

### Progressive Rollout

<div class="card-grid-2col">
  <div class="card card-accent-green">
    <h3>Week 1: Report Only</h3>
    <p class="desc">Start with <code>create-issue</code> instead of <code>create-pull-request</code>. Review reports to calibrate the agent's judgment.</p>
  </div>
  <div class="card card-accent-blue">
    <h3>Week 2: Draft PRs</h3>
    <p class="desc">Switch to <code>create-pull-request</code> with <code>draft: true</code>. Review every PR. Merge the good ones, close the bad ones.</p>
  </div>
</div>

Software development is a human endeavor, and maintaining team morale is crucial. Continuous Team Motivation uses AI to turn PRs and other team activity into poetry, zines, podcasts, provide nudges, or celebrate team achievements.

<div class="card-grid-2col">
  <div class="card card-accent-yellow">
    <h3>🏆 Milestone Celebrations</h3>

  </div>
  <div class="card card-accent-purple">
    <h3>🎨 Creative Summaries</h3>
    <p class="desc">Turns dry weekly changelogs into engaging formats like short poems, team zines, or even scripts for internal podcasts.</p>
  </div>
</div>

### How It Works

<div class="flow-diagram">
  <div class="flow-step"><span class="step-icon">🎉</span><span class="step-label">Event</span><span class="step-desc">PR merged / Milestone</span></div>
  <span class="flow-arrow">→</span>
  <div class="flow-step"><span class="step-icon">🧠</span><span class="step-label">Analyze</span><span class="step-desc">Extracts key achievements</span></div>
  <span class="flow-arrow">→</span>
  <div class="flow-step"><span class="step-icon">🎨</span><span class="step-label">Create</span><span class="step-desc">Generates creative content</span></div>
