<section id="philosophy">
<div class="section-label" style="color: var(--color-green);">Foundation</div>

## Core Security Philosophy

Four principles guide every design decision in the gh-aw security architecture.

<div class="grid-4">
  <div class="card">
    <div style="font-size: 32px; margin-bottom: 12px;">🛡️</div>
    <h3>Defense in Depth</h3>
    <p>Multiple independent layers — substrate isolation, configuration hardening, and runtime planning — each enforce distinct security properties independently.</p>
  </div>
  <div class="card">
    <div style="font-size: 32px; margin-bottom: 12px;">🔑</div>
    <h3>Zero-Secret Agents</h3>
    <p>Agents have <strong style="color: var(--color-red);">zero access</strong> to tokens, keys, or credentials. LLM keys live in an isolated API Proxy; MCP credentials in a separate Gateway. Architectural boundary, not policy.</p>
  </div>
  <div class="card">
    <div style="font-size: 32px; margin-bottom: 12px;">📋</div>
    <h3>Stage &amp; Vet All Writes</h3>
    <p>Every write is buffered as a structured artifact, analyzed by deterministic filters and a security-focused AI, then executed by separate permission-scoped jobs.</p>
  </div>
  <div class="card">
    <div style="font-size: 32px; margin-bottom: 12px;">📊</div>
    <h3>Log Everything</h3>
    <p>Comprehensive observability at every trust boundary enables forensic reconstruction and anomaly detection. Logging is a structural feature, not optional instrumentation.</p>
  </div>
</div>

<div class="bq">
  "Our first mitigation was to isolate the agent in a dedicated container with tightly controlled egress: firewalled internet access, MCP access through a trusted MCP gateway, and LLM API calls through an API proxy."
  <cite>— Landon Cox &amp; Jiaxiao Zhou, GitHub Engineering Blog</cite>
</div>
</section>

<section id="threats">
<div class="section-label" style="color: var(--color-red);">Threat Model</div>

## What the Adversary Can Do

Two properties fundamentally change the threat model: agents reason autonomously over untrusted inputs, and GitHub Actions provides a shared, highly-permissive trust domain.

<div class="grid-2" style="margin-bottom: 28px;">
  <div class="callout threat">
    <div class="callout-icon">⚠️</div>
    <div>
      <h4>Access &amp; Corrupt Shared State</h4>
      <p>A compromised agent can read environment variables, config files, SSH keys, <code>/proc</code> state, and workflow logs to discover credentials and sensitive data.</p>
    </div>
  </div>
  <div class="callout threat">
    <div class="callout-icon">⚠️</div>
    <div>
      <h4>Communicate Over Unintended Channels</h4>
      <p>Encode secrets in PR descriptions, issue bodies, comments, or other public-facing objects to exfiltrate data through legitimate GitHub features.</p>
    </div>
  </div>
  <div class="callout threat">
    <div class="callout-icon">⚠️</div>
    <div>
      <h4>Abuse Legitimate Channels</h4>
      <p>Spam issues and PRs to overwhelm maintainers, inject objectionable content, or trigger cost escalation through excessive API usage.</p>
    </div>
  </div>
  <div class="callout threat">
    <div class="callout-icon">⚠️</div>
    <div>
      <h4>Confuse Control Logic</h4>
      <p>Deviate from expected workflows, manipulate upstream decision-making, or exploit prompt injection via crafted issues, comments, and hidden Unicode.</p>
    </div>
  </div>
</div>

<div class="section-label" style="color: var(--color-orange); margin-top: 36px;">Risk Classes</div>

### Three Primary Risk Classes

<div class="grid-3">
  <div class="card" style="border-color: rgba(248,81,73,0.3);">
    <span class="badge badge-red">Critical</span>
    <h3 style="margin-top: 12px;">Data Exfiltration</h3>
    <p>Agent leaks context — including tokens — to unintended destinations via network calls or encoded in public-facing objects like PR descriptions.</p>
  </div>
  <div class="card" style="border-color: rgba(210,153,34,0.3);">
    <span class="badge badge-orange">High</span>
    <h3 style="margin-top: 12px;">Impersonation</h3>
    <p>Unclear who issued a directive or who is accountable for agent actions. Without proper attribution, malicious actions can be concealed.</p>
  </div>
  <div class="card" style="border-color: rgba(210,153,34,0.3);">
    <span class="badge badge-orange">High</span>
    <h3 style="margin-top: 12px;">Prompt Injection</h3>
    <p>Malicious users craft inputs — issues, files, comments with hidden Unicode — that trick agents into performing unauthorized behavior.</p>
  </div>
</div>

<section id="layers">
<div class="section-label" style="color: var(--color-blue);">Architecture</div>

## Three-Layer Defense in Depth

Each layer enforces independent security properties and constrains the impact of failures in the layers above it.

<div class="layer-stack">
  <div class="layer-box layer-planning">
    <span class="layer-tag">Layer 3</span>
    <h3>🧠 Planning Layer</h3>
    <p>Constrains behavior <strong style="color: var(--color-text);">over time</strong>. SafeOutputs MCP buffers writes as artifacts. Content sanitization strips injection vectors. Threat detection AI analyzes all outputs. Secret redaction scrubs credentials. GitHub Lockdown Mode filters untrusted input in public repos.</p>
  </div>
  <div class="layer-box layer-config">
    <span class="layer-tag">Layer 2</span>
    <h3>⚙️ Configuration Layer</h3>
    <p>Declarative artifacts and a trusted compiler produce security-hardened <code>.lock.yml</code> workflows. Schema validation, expression allowlists, Action SHA pinning via <code>actions-lock.json</code>, and security scanners (actionlint, zizmor, poutine) all run at compile time. Strict mode adds further constraints.</p>
  </div>
  <div class="layer-box layer-substrate">
    <span class="layer-tag">Layer 1</span>
    <h3>🏗️ Substrate Layer</h3>
    <p>Kernel-enforced isolation via Actions runner VM, Docker containers, iptables, and cgroups. Three privileged containers — <strong style="color: var(--color-text);">Network Firewall (AWF)</strong>, <strong style="color: var(--color-text);">API Proxy</strong>, and <strong style="color: var(--color-text);">MCP Gateway</strong> — mediate all external communication. Holds <em>even if the agent container is fully compromised</em>.</p>
  </div>
</div>

<div class="callout protect" style="margin-top: 28px; max-width: 820px; margin-left: auto; margin-right: auto;">
  <div class="callout-icon">✅</div>
  <div>
    <h4>Critical Guarantee</h4>
    <p>Substrate protections hold even if the agent executes arbitrary code. A breach at this layer would require vulnerabilities in the container runtime, kernel, hypervisor, or hardware.</p>
  </div>
</div>
</section>

<section id="safe-outputs">
<div class="section-label" style="color: var(--color-green);">Write Control</div>

## Safe Outputs Pipeline

All write operations follow a staged pipeline: buffer → detect → apply. The agent never writes directly to external state.

<div class="pipeline">
  <div class="pipeline-step" style="border-color: rgba(88,166,255,0.3);">
    <div class="step-num" style="background: var(--color-blue-dim); color: var(--color-blue);">1</div>
    <h4 style="color: var(--color-blue);">Buffer</h4>
    <p>Agent runs with <strong>read-only</strong> permissions. Write actions (create issue, open PR) are buffered as structured JSON artifacts — never executed immediately.</p>
    <div style="margin-top: 10px;"><span class="badge badge-blue">Read-Only Perms</span></div>
  </div>
  <div class="pipeline-arrow">→</div>
  <div class="pipeline-step" style="border-color: rgba(210,153,34,0.3);">
    <div class="step-num" style="background: var(--color-orange-dim); color: var(--color-orange);">2</div>
    <h4 style="color: var(--color-orange);">Detect</h4>
    <p>Buffered artifacts pass through deterministic filters and a <strong>security-focused AI agent</strong>. Threat detection emits a pass/fail verdict that gates all writes.</p>
    <div style="margin-top: 10px;"><span class="badge badge-orange">No Write Perms</span></div>
  </div>
  <div class="pipeline-arrow">→</div>
  <div class="pipeline-step" style="border-color: rgba(63,185,80,0.3);">
    <div class="step-num" style="background: var(--color-green-dim); color: var(--color-green);">3</div>
    <h4 style="color: var(--color-green);">Apply</h4>
    <p>Only artifacts that survive the full pipeline are applied by separate, <strong>permission-scoped</strong> jobs — create_issue, add_comment, create_pr, add_labels.</p>
    <div style="margin-top: 10px;"><span class="badge badge-green">Scoped Write Perms</span></div>
  </div>
</div>

<div style="max-width: 900px; margin: 36px auto 0;">

### Three Categories of Control

<div class="card table-wrap">

| Control | Description | Example |
|---------|-------------|---------|
| Operation Allowlisting | Authors specify which write operations an agent may perform | `create-issue`, `create-pull-request`, `add-comment` |
| Volume Limits | Cap the number of allowed operations per run | At most 3 PRs per run |
| Content Sanitization | Analyze and filter output content | Remove untrusted URLs, potential secrets, objectionable content |

</div>
</div>

<div class="callout info" style="margin-top: 24px; max-width: 900px; margin-left: auto; margin-right: auto;">
  <div class="callout-icon">💡</div>
  <div>
    <h4>Missing-Data Safe Output</h4>
    <p>A built-in safe output called <code>missing-data</code> encourages agents to explicitly report when they lack data to complete a task, rather than hallucinating. This rewards honest agent behavior and creates a feedback loop for improvement.</p>
  </div>
</div>
</section>

<section id="firewall">
<div class="section-label" style="color: var(--color-green);">Network Isolation</div>

## Agent Workflow Firewall (AWF)

Open-source network firewall (<a href="https://github.com/github/gh-aw-firewall" target="_blank">github/gh-aw-firewall</a>) that containerizes the agent, isolates the network, and routes all traffic through a Squid proxy with configurable domain allowlists.

<div class="fw-diagram">
  <div class="fw-box fw-agent">
    <h4>🤖 Agent Container</h4>
    <p style="color: var(--color-text-muted);">Isolated network<br/>namespace</p>
  </div>
  <div class="fw-arrow">→</div>
  <div class="fw-box fw-proxy">
    <h4>🔀 Squid Proxy</h4>
    <p style="color: var(--color-text-muted);">Domain allowlist<br/>at 172.30.0.10</p>
  </div>
  <div class="fw-arrow">→</div>
  <div style="display: flex; flex-direction: column; gap: 8px;">
    <div class="fw-box fw-allowed">
      <h4>✅ Allowed</h4>
      <p style="color: var(--color-text-muted);">Traffic proceeds</p>
    </div>
    <div class="fw-box fw-blocked">
      <h4>🚫 Blocked</h4>
      <p style="color: var(--color-text-muted);">Dropped &amp; logged</p>
    </div>
  </div>
</div>

<div class="grid-2" style="margin-top: 28px;">
  <div>
    <h3 style="font-size: 1rem; font-weight: 600; margin-bottom: 16px;">How It Works</h3>
    <ol style="color: var(--color-text-muted); font-size: 0.9rem; padding-left: 20px; display: flex; flex-direction: column; gap: 10px;">
      <li>AWF establishes an isolated network namespace with a Squid proxy</li>
      <li>iptables rules redirect all HTTP/HTTPS traffic through the proxy</li>
      <li>The proxy evaluates each request against the domain allowlist</li>
      <li>Allowed traffic proceeds; blocked traffic is dropped and logged</li>
      <li><strong style="color: var(--color-green);">AWF drops its iptables capabilities</strong> before launching the agent, preventing network reconfiguration</li>
    </ol>
  </div>
  <div>
    <h3 style="font-size: 1rem; font-weight: 600; margin-bottom: 16px;">Configuration</h3>
    <div class="code-block"><span class="kw">network</span>:{'\n'}  <span class="kw">firewall</span>: <span class="str">true</span>{'\n'}  <span class="kw">allowed</span>:{'\n'}    - <span class="str">defaults</span>        <span class="cmt"># Certificates, JSON schema</span>{'\n'}    - <span class="str">python</span>          <span class="cmt"># PyPI, Conda</span>{'\n'}    - <span class="str">node</span>            <span class="cmt"># npm, npmjs.com</span>{'\n'}    - <span class="str">"api.example.com"</span>  <span class="cmt"># Custom domain</span></div>

### Three-Container Design

<div class="container-grid">
  <div class="container-card-sm">
    <div class="cc-icon">🔀</div>
    <h4>Squid Proxy</h4>
    <p>Filters all outbound traffic by domain allowlist. Supports SSL Bump for HTTPS content inspection and URL path filtering.</p>
  </div>
  <div class="container-card-sm">
    <div class="cc-icon">🤖</div>
    <h4>Agent Container</h4>
    <p>Runs the command with all HTTP/HTTPS routed through Squid. Cannot reconfigure iptables — capabilities are dropped pre-launch.</p>
  </div>
  <div class="container-card-sm">
    <div class="cc-icon">🔑</div>
    <h4>API Proxy Sidecar</h4>
    <p>Holds LLM API keys so they never reach the agent process. Injects credentials into outbound requests transparently.</p>
  </div>
</div>

{/* ═══════ 7. API PROXY & MCP GATEWAY ═══════ */}

<section id="token-isolation">
<div class="section-label" style="color: var(--color-green);">Token Isolation</div>

## API Proxy &amp; MCP Gateway

Secrets never enter the agent container. Two trusted sidecar containers hold all authentication material and mediate every external call.

<div class="grid-2">
  <div class="card" style="border-top: 3px solid var(--color-green);">
    <h3 style="color: var(--color-green);">🔑 API Proxy</h3>
    <p style="margin-bottom: 14px;">A trusted sidecar that holds agent authentication tokens (LLM API keys) so they never enter the agent container.</p>
    <ul style="color: var(--color-text-muted); font-size: 0.88rem; padding-left: 18px; display: flex; flex-direction: column; gap: 6px;">
      <li>Agents route model traffic through the proxy</li>
      <li>Proxy injects credentials into outbound requests</li>
      <li>Agent sees only the proxy endpoint — never actual API keys</li>
      <li>All request/response metadata, token usage, and details are logged</li>
    </ul>
  </div>
  <div class="card" style="border-top: 3px solid var(--color-purple);">
    <h3 style="color: var(--color-purple);">🌐 MCP Gateway</h3>
    <p style="margin-bottom: 14px;">A trusted container managing communication between the agent and MCP (Model Context Protocol) servers.</p>
    <ul style="color: var(--color-text-muted); font-size: 0.88rem; padding-left: 18px; display: flex; flex-direction: column; gap: 6px;">
      <li>Spawns isolated MCP server containers — prevents cross-server attacks</li>
      <li>Holds MCP authentication material outside agent boundary</li>
      <li>Logs every tool invocation: name, arguments, result, timing</li>
      <li>Routes all MCP requests through a unified HTTP endpoint</li>
    </ul>
  </div>
</div>

<div class="callout protect" style="margin-top: 24px;">
  <div class="callout-icon">🛡️</div>
  <div>
    <h4>Why This Matters</h4>
    <p>Prompt-injected agents with shell access could read config files, SSH keys, <code>/proc</code> state, and workflow logs to discover credentials. The API Proxy and MCP Gateway eliminate this entire attack surface by keeping secrets outside the agent's isolation boundary.</p>
  </div>
</div>
</section>

## Content Sanitization Pipeline

User-generated content is sanitized before the agent ever sees it, neutralizing injection vectors at the activation boundary.

<div class="card table-wrap">

| Mechanism | Example | Protection |
|-----------|---------|------------|
| @mention Neutralization | `@user` → `` `@user` `` | Prevents unintended notifications |
| Bot Trigger Protection | `fixes #123` → `` `fixes #123` `` | Prevents automatic issue linking |
| XML/HTML Tag Conversion | `<script>` → `(script)` | Prevents injection via HTML tags |
| URI Filtering | `http://evil.com` → `(redacted)` | Restricts to HTTPS from trusted domains |
| Unicode Normalization | Homoglyphs → normalized forms | Prevents visual spoofing attacks |
| Content Limits | Large payloads → truncated | Enforces 0.5 MB max, 65k lines max |
| Control Character Removal | ANSI escapes → stripped | Removes terminal manipulation codes |

{/* ═══════ 10. GITHUB'S 6 RULES ═══════ */}

<section id="six-rules">
<div class="section-label" style="color: var(--color-orange);">Principles</div>

## GitHub's 6 Rules for Agentic Security

From <a href="https://github.blog/ai-and-ml/github-copilot/how-githubs-agentic-security-principles-make-our-ai-agents-as-secure-as-possible/" target="_blank">Rahul Zhade's blog post</a> (November 2025) — six rules that apply across all of GitHub's hosted agentic products.

<div style="max-width: 860px;">
  <div class="rule-item">
    <div class="rule-num">1</div>
    <div>
      <h4>Ensure All Context Is Visible</h4>
      <p>Display context source files and strip invisible Unicode/HTML before passing to the agent. Users must be able to see exactly what the agent sees.</p>
    </div>
  </div>
  <div class="rule-item">
    <div class="rule-num">2</div>
    <div>
      <h4>Firewall the Agent</h4>
      <p>AWF limits network access with configurable domain allowlists. Agents cannot reach arbitrary endpoints — all traffic is proxied and filtered.</p>
    </div>
  </div>
  <div class="rule-item">
    <div class="rule-num">3</div>
    <div>
      <h4>Limit Access to Sensitive Information</h4>
      <p>Only provide information absolutely necessary for the task. CI secrets are excluded. Authentication tokens are revoked after each session completes.</p>
    </div>
  </div>
  <div class="rule-item">
    <div class="rule-num">4</div>
    <div>
      <h4>Prevent Irreversible State Changes</h4>
      <p>Agents create pull requests — not direct commits. CI doesn't auto-run on agent PRs. A human must validate the code and manually trigger workflows.</p>
    </div>
  </div>
  <div class="rule-item">
    <div class="rule-num">5</div>
    <div>
      <h4>Consistently Attribute Actions</h4>
      <p>Actions are co-committed by the initiating user. The agent identity (Copilot) is clearly marked on all PRs. Accountability chains are always visible.</p>
    </div>
  </div>
  <div class="rule-item">
    <div class="rule-num">6</div>
    <div>
      <h4>Only Gather Context from Authorized Users</h4>
      <p>Agents can only be assigned by users with write access. In public repos, agents only read comments from users with write access — preventing context poisoning.</p>
    </div>
  </div>
</div>
</section>

<div class="section-label" style="color: var(--color-green);">Summary</div>

## Complete Security Layers

Every layer from kernel isolation to AI-powered detection working together to ensure automation scales without scaling risk.

<div class="card table-wrap">

| Layer | Mechanism | Protects Against |
|-------|-----------|------------------|
| <span class="badge badge-green">Substrate</span> | Actions runner VM (kernel, hypervisor) | Memory corruption, privilege escalation, host escape |
| <span class="badge badge-green">Substrate</span> | Docker container isolation | Process isolation bypass, shared state access |
| <span class="badge badge-green">Substrate</span> | AWF network controls (iptables + Squid) | Data exfiltration, unauthorized API calls |
| <span class="badge badge-green">Substrate</span> | MCP sandboxing (per-server containers) | Cross-server attacks, unauthorized tool access |
| <span class="badge badge-purple">Config</span> | Schema validation, expression allowlist | Invalid configurations, secret exposure |
| <span class="badge badge-purple">Config</span> | Action SHA pinning | Supply-chain attacks, tag hijacking |
| <span class="badge badge-purple">Config</span> | Security scanners (actionlint, zizmor, poutine) | Privilege escalation, misconfigurations |
| <span class="badge badge-blue">Plan</span> | Content sanitization &amp; Lockdown Mode | @mention abuse, injection, context poisoning |
| <span class="badge badge-blue">Plan</span> | Secret redaction (pre-upload) | Credential leakage in logs/artifacts |
| <span class="badge badge-blue">Plan</span> | AI-powered threat detection | Malicious patches, secret leaks |
| <span class="badge badge-blue">Plan</span> | SafeOutputs permission separation | Direct write access abuse |
