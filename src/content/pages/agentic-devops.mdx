## How We Got Here

<p class="section-desc">Every era of software delivery follows the same pattern: compress the feedback loop. The tighter the loop, the faster you learn, the lower the risk, the more you ship. GitHub Agentic Workflows isn't just a technical implementation — it's the foundation of a new branch of DevOps.</p>

## Layer 0 — The Sandbox

<p class="section-desc">Other agent frameworks give you hooks and guidelines. GitHub Agentic Workflows gives you a <strong style="color: var(--color-text);">kernel-level execution boundary</strong> beneath everything else. Without isolation, governance is advisory. With it, governance is structural. This is what makes agentic workflows shine.</p>

<div class="sandbox-grid">
  <div class="sandbox-card">
    <div class="sandbox-icon">📦</div>
    <h3>Agent Sandboxes</h3>
    <p>Policy-governed isolation environments where agents execute safely — filesystem, network, and process boundaries enforced at the kernel level, not application level.</p>
  </div>
  <div class="sandbox-card">
    <div class="sandbox-icon">🔒</div>
    <h3>Network Gating</h3>
    <p>Deny-by-default outbound access. Agents can only reach explicitly whitelisted endpoints — declared in YAML, enforced by proxy. No phoning home, no data exfiltration.</p>
  </div>
  <div class="sandbox-card">
    <div class="sandbox-icon">🛡️</div>
    <h3>Credential Isolation</h3>
    <p>API keys injected at runtime, never on disk. Inference routing strips agent credentials and injects backend keys — context never leaks to the model.</p>
  </div>
</div>
<div class="sandbox-callout">
  <p>💡 <strong>Why Layer 0 matters:</strong> Hooks and instructions are speed bumps — agents can misinterpret or ignore them. A sandbox is a wall. When the execution boundary is structural, every layer above it — deterministic gates, enablement context, CI/CD pipelines — can trust that the agent <em>cannot</em> escape its constraints, not just that it <em>shouldn't</em>.</p>
</div>

<section id="pillars">
<div class="section-label">The Vision</div>

## The Six Pillars of Continuous AI

<p class="section-desc">Continuous AI is the systematic, automated application of AI to software collaboration — natural-language rules combined with agentic reasoning, executed continuously inside your repository. It augments CI/CD for judgment-heavy, context-dependent tasks.</p>

<div class="pillars-grid">
  <div class="pillar-card">
    <div class="pillar-icon">🏷️</div>
    <h3>Continuous Triage</h3>
    <p>Automatically summarize, label, and route new issues. The agent understands context and intent — far beyond keyword matching.</p>
  </div>
  <div class="pillar-card">
    <div class="pillar-icon">📝</div>
    <h3>Continuous Documentation</h3>
    <p>Keep READMEs and docs aligned with code changes. Detect mismatches between docstrings and implementations, then open PRs to fix them.</p>
  </div>
  <div class="pillar-card">
    <div class="pillar-icon">🧹</div>
    <h3>Continuous Code Simplification</h3>
    <p>Identify refactoring opportunities — dead code, complex conditionals, repeated patterns — and propose targeted improvements via PRs.</p>
  </div>
  <div class="pillar-card">
    <div class="pillar-icon">🧪</div>
    <h3>Continuous Test Improvement</h3>
    <p>Assess test coverage and add high-value tests. One experiment: 5% → near 100% coverage, 1,400+ tests written over 45 days for ~$80.</p>
  </div>
  <div class="pillar-card">
    <div class="pillar-icon">🔧</div>
    <h3>Continuous Quality Hygiene</h3>
    <p>Proactively investigate CI failures and propose targeted fixes — flaky tests, dependency conflicts, and configuration issues.</p>
  </div>
  <div class="pillar-card">
    <div class="pillar-icon">📊</div>
    <h3>Continuous Reporting</h3>
    <p>Generate regular reports on repository health, synthesizing data from issues, PRs, commits, and CI into actionable insights.</p>
  </div>
</div>
