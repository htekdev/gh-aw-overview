## 🤖 Copilot Coding Agent

<p class="section-desc">The autonomous runtime engine that executes agentic workflows in ephemeral GitHub Actions environments.</p>

<div class="grid-2">
  <div class="card">
    <h3>What It Does</h3>
    <p>The Copilot coding agent is an autonomous AI developer that works independently in the background. It operates in ephemeral <strong>GitHub Actions</strong> environments where it can explore code, make changes, run tests, iterate on failures, and open pull requests.</p>
    <ul style="margin-top: 12px;">
      <li>Fix bugs identified in issues or code review</li>
      <li>Implement incremental features from scoped descriptions</li>
      <li>Improve test coverage by writing missing tests</li>
      <li>Address technical debt (refactoring, dependency updates)</li>
      <li>Update documentation to match code changes</li>
      <li>Accelerate secret scanning remediation</li>
    </ul>
  </div>
  <div class="card">
    <h3>Key Insight</h3>
    <div class="callout" style="margin-top: 0;">
      <strong>CCA ≠ Agent Mode.</strong> The Copilot coding agent works <em>autonomously</em> in a GitHub Actions environment and creates PRs. Agent mode in the IDE works <em>interactively</em> in your local environment with human-in-the-loop control.
    </div>
    <h3 style="margin-top: 16px;">Session Management</h3>
    <ul>
      <li><strong>Steering</strong> — Provide mid-session guidance without stopping the run</li>
      <li><strong>Resume locally</strong> — Open in VS Code or paste a CLI command to continue</li>
      <li><strong>Archiving</strong> — Stopped sessions can be archived</li>
      <li><strong>Tracking</strong> — Monitor from CLI, Mobile, VS Code, Raycast, JetBrains</li>
    </ul>
  </div>
</div>

<div class="grid-2">
  <div class="card border-blue">
    <h3>Path-Specific Instructions</h3>
    <p>Use YAML frontmatter with an <code>applyTo</code> glob pattern to scope instructions to specific file types:</p>
  </div>
  <div class="card border-blue">
    <h3>AGENTS.md</h3>
    <p>Added to the coding agent in <strong>August 2025</strong>. Key characteristics:</p>
    <ul>
      <li>Placed <strong>anywhere</strong> in the repository</li>
      <li>Nearest <code>AGENTS.md</code> in directory tree takes precedence</li>
      <li>Subdirectory files scope to that directory and children</li>
      <li>Root <code>AGENTS.md</code> applies globally</li>
      <li>Follows the open <a href="https://github.com/agentsmd/agents.md">agentsmd spec</a></li>
      <li><code>CLAUDE.md</code> and <code>GEMINI.md</code> are read <em>in addition</em></li>
    </ul>
  </div>
</div>

Path-specific instructions example:

```yaml
---
applyTo: "**/*.bicep"
---
When writing Bicep files, always use parameter
decorators, follow naming conventions, and include
resource descriptions for documentation.
```

<div class="card border-blue" style="margin-top: 4px;">
  <h3>Precedence Model</h3>
</div>

```
Layer 0: Personal Instructions (VS Code Settings)          ← Highest priority
Layer 1: copilot-instructions.md (repo-wide)                ← Always-on
Layer 2: *.instructions.md (matched by applyTo glob)        ← Conditionally loaded
Layer 3: AGENTS.md (when coding agent is operating)         ← Agent-scoped
Layer 4: Skill (if the task matches a skill description)    ← On-demand
Layer 5: Custom Agent prompt (if a specific agent selected) ← Explicitly selected
```

All applicable instructions are provided simultaneously — contradictions between layers should be avoided.

<div class="callout" style="margin-top: 16px;">
  <strong>Best practice:</strong> Keep <code>copilot-instructions.md</code> concise (5–15 core rules, 200–500 tokens). Use path-specific instructions for language-specific conventions. Reserve <code>AGENTS.md</code> for autonomous agent operational procedures.
</div>

<section id="prompts">

## 📄 Prompt Files

<p class="section-desc">Reusable, named tasks invoked on demand as slash commands — bridging always-on instructions and full custom agents.</p>

<div class="grid-2">
  <div class="card border-green">
    <h3>Format &amp; Configuration</h3>
    <ul>
      <li>Markdown files in <code>.github/prompts/&#42;.prompt.md</code></li>
      <li>Invoked with <code>/</code> in Copilot Chat (e.g., <code>/security-review</code>)</li>
      <li>Support YAML frontmatter: <code>agent</code>, <code>tools</code>, <code>model</code>, <code>description</code>, <code>mode</code></li>
      <li>Built-in variables: <code>$&#123;selection&#125;</code>, <code>$&#123;file&#125;</code>, <code>$&#123;input:variableName&#125;</code></li>
      <li>Can reference a custom agent via the <code>agent</code> field</li>
    </ul>
  </div>
  <div class="card border-green">
    <h3>Example</h3>
  </div>
</div>

```yaml
---
mode: agent
tools: ['search/codebase', 'edit/editFiles',
        'read/problems']
description: Create a structured implementation
             plan for a new feature
---
# Feature Planning

You are a senior software architect.
Create an implementation plan for the
following feature: ${input:featureDescription}

## Requirements
- Break down into discrete tasks
- Identify affected files using codebase search
- Estimate complexity per task
```

<div class="callout callout-green">
  <strong>When to use Prompt Files:</strong> Repeatable tasks you type the same prompt for, tasks benefiting from dynamic variables, and named slash commands the whole team can share via source control.
</div>

</section>

### Collaboration Patterns

<div class="grid-2">
  <div class="card" style="border-left: 4px solid var(--color-purple); border-top: 3px solid var(--color-purple);">
    <h3>🤝 Handoffs <span class="badge badge-purple">User-Guided</span></h3>
    <p>Relay-race style transitions where the user explicitly confirms each switch. Shared conversation history across the chain.</p>
  </div>
  <div class="card" style="border-left: 4px solid var(--color-orange); border-top: 3px solid var(--color-orange);">
    <h3>🔄 Sub-agents <span class="badge badge-orange">Programmatic</span></h3>
    <p>Manager-delegate style where the parent agent autonomously invokes child agents without user intervention.</p>
  </div>
</div>

Handoffs example:

```yaml
---
name: Planner
description: Generate implementation plans
tools: ['search', 'fetch']
handoffs:
  - label: Start Implementation
    agent: Implementer
    prompt: "Implement the plan above."
    send: false  # User must click
---
```

**Handoff flow:** Planner → *\[User clicks\]* → Implementer → *\[User clicks\]* → Reviewer

- Human-in-the-loop control
- Shared conversation history
- Unlimited depth

Sub-agents example:

```yaml
---
name: Feature Builder
description: Build features by researching
             first, then implementing
tools: ['agent']
agents: ['Researcher', 'Implementer']
---
```

**Sub-agent flow:** Feature Builder autonomously delegates → Researcher → Implementer

- Agent-autonomous (no user clicks)
- Isolated context (clean session)
- Single level depth only

<div class="table-wrap" style="margin-top: 20px;">
  <table>
    <thead><tr><th>Dimension</th><th>Handoffs</th><th>Sub-agents</th></tr></thead>
    <tbody>
      <tr><td><strong>Trigger</strong></td><td>User clicks a button</td><td>Parent agent invokes programmatically</td></tr>
      <tr><td><strong>Control</strong></td><td>User-driven (human-in-the-loop)</td><td>Agent-autonomous</td></tr>
      <tr><td><strong>Context</strong></td><td>Shared conversation history</td><td>Isolated context (clean session)</td></tr>
      <tr><td><strong>Visibility</strong></td><td>User sees the agent switch</td><td>Transparent — user sees only the result</td></tr>
      <tr><td><strong>Depth</strong></td><td>Unlimited</td><td>Single level only</td></tr>
      <tr><td><strong>Best for</strong></td><td>Multi-stage pipelines needing human review</td><td>Automated divide-and-conquer tasks</td></tr>
    </tbody>
  </table>
</div>

### MCP Servers in Agent Profiles

<div class="card border-purple">
  <p>Custom agents can embed MCP server configurations directly in their frontmatter:</p>
</div>

```yaml
---
name: my-custom-agent-with-mcp
description: Custom agent description
tools: ['tool-a', 'tool-b', 'custom-mcp/tool-1']
mcp-servers:
  custom-mcp:
    type: 'local'
    command: 'some-command'
    args: ['--arg1', '--arg2']
    tools: ["*"]
    env:
      ENV_VAR_NAME: ${{ secrets.COPILOT_MCP_ENV_VAR_VALUE }}
---
```

Secret syntax supports: `$COPILOT_MCP_*`, `${COPILOT_MCP_*}`, and `${{ secrets.COPILOT_MCP_* }}`.

## 📋 Agent Skills

<p class="section-desc">The workflow packaging layer — reusable runbooks that bundle instructions, scripts, templates, and reference materials into self-contained capability packages.</p>

<div class="callout callout-orange">
  <strong>Open Standard:</strong> Agent Skills follow the <a href="https://agentskills.io">agentskills.io</a> specification (originally developed by Anthropic). A skill defined once works in GitHub Copilot, Claude Code, and any other skills-compatible agent.
</div>

### Progressive Loading Model (Three-Tier)

<div class="table-wrap">
  <table>
    <thead><tr><th>Tier</th><th>Content</th><th>Token Budget</th><th>When Loaded</th></tr></thead>
    <tbody>
      <tr>
        <td><span class="badge badge-green">Tier 1</span> Metadata</td>
        <td><code>name</code> + <code>description</code> (YAML frontmatter)</td>
        <td>~50–100 tokens/skill</td>
        <td><strong>Always</strong> in context</td>
      </tr>
      <tr>
        <td><span class="badge badge-blue">Tier 2</span> Body</td>
        <td>SKILL.md body content</td>
        <td>Recommended &lt; 500 tokens</td>
        <td>When Copilot determines relevance</td>
      </tr>
      <tr>
        <td><span class="badge badge-purple">Tier 3</span> References</td>
        <td>Files under <code>references/</code>, <code>scripts/</code>, etc.</td>
        <td>Recommended &lt; 1,000 tokens/file</td>
        <td>When explicitly referenced</td>
      </tr>
    </tbody>
  </table>
</div>

At startup, agents load **only the name and description** of each available skill — just enough to know when it might be relevant. Full content is loaded progressively.

<div class="grid-2">
  <div class="card border-orange">
    <h3>Directory Structure</h3>
  </div>
  <div class="card border-orange">
    <h3>SKILL.md Format</h3>
  </div>
</div>

Directory structure:

```
.github/skills/incident-triage/
├── SKILL.md              # Required: primary instructions
├── LICENSE.txt            # Recommended: license terms
├── scripts/               # Optional: executable scripts
│   ├── collect-metrics.sh
│   └── notify-oncall.ps1
├── references/            # Optional: on-demand docs
│   ├── runbook.md
│   └── escalation-matrix.md
├── assets/                # Optional: static resources
│   └── triage-template.md
└── templates/             # Optional: scaffolds
    └── incident-report.md
```

SKILL.md format:

```yaml
---
name: github-actions-failure-debugging
description: >
  Guide for debugging failing GitHub Actions
  workflows. Use this when asked to debug
  failing GitHub Actions workflows.
---

To debug failing GitHub Actions workflows:

1. Use `list_workflow_runs` to look up
   recent runs and their status
2. Use `summarize_job_log_failures` for
   an AI summary of failed jobs
3. Use `get_job_logs` for full failure
   logs if needed
4. Reproduce locally, then fix and verify
```

<div class="callout callout-pink">
  <strong>Critical:</strong> The <code>description</code> field is the <em>sole criterion</em> Copilot uses to determine whether to load a skill. Include both what-it-does and when-to-use-it.
</div>

### Skills vs Instructions

<div class="table-wrap">
  <table>
    <thead><tr><th>Concern</th><th>Instructions</th><th>Skills</th></tr></thead>
    <tbody>
      <tr><td><strong>Loading</strong></td><td>Always-on or pattern-matched</td><td>On-demand (description-matched)</td></tr>
      <tr><td><strong>Assets</strong></td><td>Text only</td><td>Scripts, templates, references</td></tr>
      <tr><td><strong>Token cost</strong></td><td>Always in context</td><td>Progressive — loaded when relevant</td></tr>
      <tr><td><strong>Portability</strong></td><td>Copilot-specific</td><td><strong>Open standard</strong> (multi-agent)</td></tr>
      <tr><td><strong>Best for</strong></td><td>Simple rules for every interaction</td><td>Detailed workflows for specific tasks</td></tr>
    </tbody>
  </table>
</div>

<p class="section-desc">Model Context Protocol — an open standard defining how applications share context with large language models, connecting AI agents to live external systems.</p>

<section id="hooks">

## 🔒 Hooks System

<p class="section-desc">The policy enforcement layer — deterministic shell commands that run at strategic points in an agent's workflow. Unlike instructions (advisory), hooks are deterministic: a deny is a deny.</p>

### Eight Hook Types

<div class="table-wrap">
  <table>
    <thead><tr><th>Hook Type</th><th>When It Fires</th><th>Key Capability</th></tr></thead>
    <tbody>
      <tr><td><code class="c-hooks">sessionStart</code></td><td>New session begins or resumes</td><td>Initialize environments, validate project state, show banners</td></tr>
      <tr><td><code class="c-hooks">sessionEnd</code></td><td>Session completes or terminates</td><td>Cleanup resources, generate reports, session summaries</td></tr>
      <tr><td><code class="c-hooks">userPromptSubmitted</code></td><td>User submits a prompt</td><td>Log requests for auditing, scan for threat signals</td></tr>
      <tr><td><code class="c-hooks" style="font-weight: 700;">preToolUse</code></td><td>Before any tool execution</td><td><strong>Approve or deny</strong> tool executions (most powerful hook)</td></tr>
      <tr><td><code class="c-hooks">postToolUse</code></td><td>After tool completes</td><td>Log results, track metrics, send alerts</td></tr>
      <tr><td><code class="c-hooks">agentStop</code></td><td>Main agent finishes responding</td><td>Post-processing, notifications</td></tr>
      <tr><td><code class="c-hooks">subagentStop</code></td><td>Sub-agent completes</td><td>Validate sub-agent results</td></tr>
      <tr><td><code class="c-hooks">errorOccurred</code></td><td>Error during agent execution</td><td>Error logging, notifications</td></tr>
    </tbody>
  </table>
</div>

<div class="grid-2" style="margin-top: 20px;">
  <div class="card border-pink">
    <h3>Configuration Format</h3>
    <p>Hooks are JSON files in <code>.github/hooks/&#42;.json</code>:</p>
  </div>
  <div class="card border-pink">
    <h3>Input / Output Format</h3>
    <p>Hooks receive detailed JSON via <strong>stdin</strong>:</p>
  </div>
</div>

Hook configuration example:

```json
{
  "version": 1,
  "hooks": {
    "sessionStart": [
      {
        "type": "command",
        "bash": "./scripts/session-banner.sh",
        "powershell": "./scripts/session-banner.ps1",
        "cwd": ".github/hooks",
        "timeoutSec": 10
      }
    ],
    "preToolUse": [
      {
        "type": "command",
        "bash": "./scripts/security-check.sh",
        "powershell": "./scripts/security-check.ps1",
        "cwd": "scripts",
        "env": {
          "LOG_LEVEL": "INFO"
        }
      }
    ]
  }
}
```

preToolUse input:

```json
{
  "timestamp": 1704614600000,
  "cwd": "/path/to/project",
  "toolName": "bash",
  "toolArgs": "{\"command\":\"rm -rf /\"}"
}
```

preToolUse output (to deny):

```json
{
  "decision": "deny",
  "reason": "Destructive command blocked"
}
```

postToolUse input includes results:

```json
{
  "toolResult": {
    "resultType": "success",
    "result": "..."
  }
}
```

<div class="table-wrap">
  <table>
    <thead><tr><th>Platform</th><th>Status</th><th>Config Location</th></tr></thead>
    <tbody>
      <tr><td><strong>Copilot coding agent</strong> (GitHub.com)</td><td><span class="badge badge-green">GA</span></td><td><code>.github/hooks/&#42;.json</code> (default branch)</td></tr>
      <tr><td><strong>GitHub Copilot CLI</strong></td><td><span class="badge badge-green">GA</span></td><td><code>.github/hooks/&#42;.json</code> (current working directory)</td></tr>
      <tr><td><strong>VS Code</strong></td><td><span class="badge badge-blue">Preview</span></td><td><code>.github/hooks/&#42;.json</code></td></tr>
    </tbody>
  </table>
</div>

### Governance Example: Comprehensive Audit Trail

```json
{
  "version": 1,
  "hooks": {
    "sessionStart":          [{ "type": "command", "bash": "./audit/log-session-start.sh" }],
    "userPromptSubmitted":   [{ "type": "command", "bash": "./audit/log-prompt.sh" }],
    "preToolUse":            [{ "type": "command", "bash": "./audit/log-tool-use.sh" }],
    "postToolUse":           [{ "type": "command", "bash": "./audit/log-tool-result.sh" }],
    "sessionEnd":            [{ "type": "command", "bash": "./audit/log-session-end.sh" }]
  }
}
```

### Governance Audit Hook (Community)

<div class="card border-pink">
  <p>The <a href="https://github.com/github/awesome-copilot/blob/main/hooks/governance-audit/README.md">governance-audit hook</a> from awesome-copilot provides:</p>
  <div class="grid-3" style="margin-top: 12px;">
    <div><strong>Threat detection</strong> — Scans for data exfiltration, privilege escalation, prompt injection</div>
    <div><strong>Governance levels</strong> — Open, Standard, Strict, Locked (allowlist-only)</div>
    <div><strong>Audit trail</strong> — Append-only JSON log; full prompts are never logged</div>
  </div>
</div>

Configure governance level in hooks.json env:

```json
{
  "env": {
    "GOVERNANCE_LEVEL": "strict",
    "BLOCK_ON_THREAT": "true"
  }
}
```

<div class="callout callout-pink" style="margin-top: 16px;">
  <strong>Performance:</strong> Hooks run <em>synchronously</em> and block agent execution — keep them under 5 seconds. Default timeout is 30s. Always validate/sanitize JSON input and never log sensitive data.
</div>

### Composition Rules

<div class="grid-2">
  <div class="card border-purple">
    <h3>Layering</h3>
    <ul>
      <li>A <strong>custom agent</strong> can reference <strong>instruction files</strong> via Markdown links</li>
      <li>A <strong>prompt file</strong> can set <code>agent:</code> to run inside a specific <strong>custom agent</strong></li>
      <li>A <strong>skill</strong> can bundle scripts that any <strong>agent</strong> or <strong>prompt file</strong> invokes</li>
      <li>An <strong>MCP server</strong> provides tools any agent, prompt, or skill can call</li>
      <li><strong>Hooks</strong> enforce policy regardless of which primitive triggered the action</li>
    </ul>
  </div>
  <div class="card border-blue">
    <h3>Decision Flow</h3>
    <ol style="padding-left: 20px;">
      <li>Need a rule on <strong>every</strong> request? → <span class="c-instructions"><strong>Instructions</strong></span></li>
      <li>Need a <strong>named, repeatable</strong> task? → <span class="c-prompts"><strong>Prompt File</strong></span></li>
      <li>Need a <strong>specialist persona</strong> with limited tools? → <span class="c-agents"><strong>Custom Agent</strong></span></li>
      <li>Need a <strong>reusable runbook</strong> with scripts? → <span class="c-skills"><strong>Skill</strong></span></li>
      <li>Need <strong>live external data</strong>? → <span class="c-mcp"><strong>+ MCP</strong></span></li>
      <li>Must be <strong>enforced deterministically</strong>? → <span class="c-hooks"><strong>+ Hook</strong></span></li>
    </ol>
    <p style="margin-top: 12px; font-size: 13px; color: var(--color-text-muted);">Questions 1–4 are "pick one" decisions. Questions 5–6 are <strong>additive layers</strong> stacked on top.</p>
  </div>
</div>
